# NewLife.XCode 竞品分析报告

## 1. 概述

本报告对比 **NewLife.XCode**（数据中间件 / 超级 ORM）与当前 .NET 生态中主流 ORM/数据访问库，从功能覆盖、性能、依赖、许可证、社区活跃度等维度进行综合评估。功能目标见[需求文档](需求文档.md)，完整功能状态见[功能模块清单](功能模块清单.md)。

XCode 定位：高性能 .NET 数据中间件，聚焦"添删改查 + 极致性能 + 海量数据"，内置多级缓存、自动建模/迁移、分表分库、强类型查询、跨库迁移，单表生产实践达百亿级。

## 5. 差距分析

### 5.1 本项目优势

| 优势领域 | 详情 |
|---------|------|
| **多级缓存体系** | EntityCache + SingleCache + Meta.Count + DbCache 四层缓存，竞品无法比拟 |
| **分表分库** | 时间分片 + 透明路由 + 跨分片查询，百亿级生产验证 |
| **国产数据库支持** | 达梦/金仓/瀚高/TDengine/InfluxDB 原生支持领先同类 |
| **企业级数据中间件** | ETL 管道、数据同步、备份恢复、实体队列、配置中心，远超纯 ORM 定位 |
| **模型驱动代码生成** | XML→代码的生成策略（数据类覆盖 + 业务类保留）设计独特且实用 |
| **内存联表策略** | 独有模式，规避复杂 JOIN 开销，适合缓存优先架构 |
| **兼容性** | net45 起兼容，MIT 协议可替换品牌私有化，商业友好度最高 |
| **生态协同** | 与 NewLife.Core/Cube/Redis/Stardust/AntJob 深度集成 |

### 5.2 待补齐差距

| 差距领域 | 优先级 | 竞品参考 | 说明 |
|---------|:---:|------|------|
| **导航属性/关联查询** | ⚠️ 部分实现 | EF Core / FreeSql | Include(Type) 预加载整表缓存（适合查找表），NavigationLoader 批量 N+1 优化，GetNavigation 惰性加载（10s 缓存）。缺失：Lambda 表达式 Include、代理延迟加载、LINQ 自动 LEFT JOIN |
| **窗口函数 / CTE** | ⚠️ 部分实现 | linq2db | WindowFunction 静态助手生成 ROW_NUMBER/RANK/DENSE_RANK/聚合窗口 SQL，未与 LINQ 表达式树集成，CTE 需手写原生 SQL |
| **社区规模和国际影响力** | 🟡 中 | EF Core / Dapper | 加强英文文档和社区推广 |

### 5.3 API 易用性对比

以下通过代码对比 XCode 与竞品在关键功能上的 API 表达差异。

#### 动态条件拼接

```csharp
// XCode：WhereExpression 或 IQueryable<T>（LINQ 风格）
// 方式一：WhereExpression 链式
var exp = new WhereExpression();
if (!name.IsNullOrEmpty()) exp &= _.UserName.Contains(name);
if (age > 0) exp &= _.Age == age;
return UserInfo.FindAll(exp, pager);

// 方式二：IQueryable<T> LINQ 查询
IQueryable<User> query = UserInfo.Query;
if (!name.IsNullOrEmpty())
    query = query.Where(u => u.UserName.Contains(name));
if (age > 0)
    query = query.Where(u => u.Age == age);
var list = query.OrderByDescending(u => u.Id)
    .Skip((page - 1) * size).Take(size)
    .ToList();

// 方式三：WhereIf 内联条件（IQueryable 扩展）
var list = UserInfo.Query
    .WhereIf(!name.IsNullOrEmpty(), u => u.UserName.Contains(name))
    .WhereIf(age > 0, u => u.Age == age)
    .Page(page, size)
    .ToList();

// EF Core / FreeSql / SqlSugar：IQueryable / WhereIf 风格与 XCode 方式二/三一致
IQueryable<User> query = ctx.Users;
if (!string.IsNullOrEmpty(name))
    query = query.Where(u => u.UserName.Contains(name));
var list = query.OrderByDescending(u => u.Id)...ToList();
```

XCode 兼容 LINQ 标准，`Entity.Query` 属性 + `WhereIf`/`Page` 扩展方法实现了与 EF Core 一致的 IQueryable 编程体验。

#### 导航属性 / 关联查询

```csharp
// XCode：三级衔接策略
// 1) Include(Type) 查询前预加载关联实体缓存
var users = UserInfo.Query.Include(typeof(Role)).Where(u => u.Enable == true).ToList();

// 2) NavigationLoader 事后批量加载（N+1 → 2 次查询）
var users = User.FindAll();
users.LoadNavigation("Role");     // 1 次批量 In 查询
users.LoadNavigation("Orders");   // 另 1 次批量 In 查询

// 3) 实体属性惰性加载（首次访问自动查库，10s 缓存）
public Role? Role => GetNavigation<Role>(nameof(Role));
public IList<Order> Orders => GetNavigationCollection<Order>(nameof(Orders));

// EF Core：Lambda Include + 代理延迟加载
var users = ctx.Users.Include(u => u.Orders).Where(u => u.Age > 18).ToList();
```

XCode 通过缓存预加载 + 批量 N+1 优化 + 惰性缓存三级策略覆盖关联查询，适合查找表为主的业务模型。EF Core 的 Lambda Include 表达式更灵活，代理延迟加载在任意深度导航时体验更统一。

## 6. 结论与建议

### 6.1 定位总结

XCode 在 .NET ORM 市场中具有**独特定位**：

- **不是又一个通用 ORM**，而是"数据中间件" — 集 ORM + 缓存 + 分片 + ETL + 同步 + 配置中心于一体
- **缓存优先架构**是核心差异化优势，在读多写少场景下的性能远超任何竞品
- **国产替代首选**：对国产数据库和时序数据库的支持领先同类，MIT 协议商业友好

### 6.2 适用场景推荐

| 场景 | 推荐 | 理由 |
|------|:---:|------|
| 读多写少、高并发查询 | ⭐⭐⭐ XCode | 多级缓存命中率极高 |
| 海量数据（亿级以上） | ⭐⭐⭐ XCode | 分表分库 + 分页优化 |
| 国产数据库项目 | ⭐⭐⭐ XCode | 国产库适配最全面 |
| 复杂多表关联查询 | ⭐⭐ EF Core / linq2db | LINQ 表达力更强 |
| 极简场景、已有 SQL | ⭐⭐ Dapper | 零依赖、极致轻量 |
| 需要 AOT 发布 | ⭐⭐ FreeSql / SqlSugar | XCode 明确不支持 AOT |
| 企业级数据平台 | ⭐⭐⭐ XCode | ETL + 同步 + 配置中心 |
