# NewLife.XCode 需求文档

> 版本：v1.1 | 日期：2026-07-01

本文档描述 NewLife.XCode 的愿景、核心目标和模块划分。完成状态在[功能模块清单](功能模块清单.md)中追踪，详细设计在各模块架构文档中展开。

## 1. 背景与愿景

### 1.1 系统定位

高性能 .NET 数据中间件（超级 ORM），聚焦"添删改查 + 极致性能 + 海量数据"，内置多级缓存、自动建模/迁移、分表分库、强类型查询、跨库迁移，支持多种关系型/时序型数据库。单表生产实践达百亿级，查询吞吐可达十亿级 QPS（配合缓存策略）。

### 1.2 愿景

成为 .NET 生态中最快速、最稳定、最易用的数据中间件——"越用越快"：写入越多，缓存命中越精准；查询越多，预热越充分。为物联网、工业互联网、金融交易等高吞吐场景提供坚实的数据底座。

## 2. M1 查询与条件 DSL

- **M1-1 WhereExpression 链式查询**：提供组合条件 &/\| 链式表达式，支持 Contains、Between、Like、In 等常见条件
- **M1-2 IQueryable&lt;T&gt; LINQ 查询**：✅ 已实现——实体级 `Query` 属性返回 `EntityQueryable<T>`（实现 `IQueryable<T>`）。基于 `EntityQueryProvider` + `LinqExpressionVisitor` 将 LINQ 表达式树翻译为 XCode WhereExpression，支持 Where（含 Contains/StartsWith）、OrderBy/ThenBy、Skip/Take、Count/First/FirstOrDefault/ToList 等标准操作。
- **M1-3 WhereIf 内联条件**：✅ 已实现——IQueryable 扩展 `WhereIf(condition, predicate)`（`LinqExtensions` 类），条件不满足时自动忽略；实体级静态方法 `FindAllWhereIf(params (bool, Expression)[])`，支持 C# 元组内联拼接。
- **M1-4 Include 预加载**：✅ 已实现——`Include(Type)` 预加载关联实体的整表缓存（EntityCache），与 XCode 内存联表策略互补。查询执行前批量预热缓存，后续内存联表直接命中。
- **M1-5 分页与排序 (Pager/PageFindAll)**：统一的分页参数对象，自动处理 TotalCount 统计、排序字段、数据权限扩展
- **M1-6 字段投影与自定义排序**：支持指定查询列、多字段排序方向
- **M1-7 原生 SQL 查询**：兼容 Dapper 用法，支持执行任意 SQL 并映射到实体
- **M1-8 多结果集**：一次查询返回多个结果集
- **M1-9 内存联表**：先命中缓存再内存关联，规避复杂 JOIN
- **M1-10 窗口函数**：✅ 已实现——SelectBuilder.Window 属性支持标准 SQL 窗口函数（ROW_NUMBER/RANK/DENSE_RANK），配合 `WindowFunction` 静态助手构建 OVER(PARTITION BY...ORDER BY...) 子句，同时支持 SUM/AVG/COUNT 等聚合窗口函数。

## 3. M2 多级缓存体系

- **M2-1 Meta.Count 行数缓存**：快速行数缓存，>1000 行走系统索引，支持异步精准补偿
- **M2-2 EntityCache 整表缓存**：适合小表（<1000 行），全量缓存到内存
- **M2-3 SingleCache 单对象缓存**：主键/从键字典级缓存，适合频繁点查
- **M2-4 DbCache 数据层缓存**：DAL 层 SQL 查询结果缓存
- **M2-5 缓存过期与刷新策略**：默认 60s 过期，过期访问返回旧数据 + 异步刷新
- **M2-6 写操作缓存一致性**：写操作即时失效/更新，事务中累积提交后统一处理
- **M2-7 二级缓存（分布式）**：✅ 已实现——`XCodeSetting.CacheProvider` 注入 Redis 等分布式 ICache，`CacheInvalidator` 基于版本号机制跨进程协调实体缓存失效。EntityCache/SingleCache 的 Clear 操作自动通知其它进程；CheckCache 定时比对远程版本号强制过期本地缓存。

## 4. M3 模型驱动与代码生成

- **M3-1 XML 模型驱动**：Tables/Columns/Indexes → 代码生成器 → 实体数据类 + 业务分部类
- **M3-2 代码生成器 (xcodetool)**：CLI 工具，数据类每次可覆盖，业务类首次生成保留
- **M3-3 反向工程 (DbFirst)**：从已有数据库抽取模型生成实体
- **M3-4 自动迁移 (CodeFirst)**：模型/实体即结构，新增字段上线自动同步，无需额外 SQL 脚本
- **M3-5 Attribute 标注**：支持 [Model] [BindIndex] 等特性标注实体元数据

## 5. M4 分表分库

- **M4-1 ShardPolicy 抽象**：统一的表/库分片策略接口
- **M4-2 TimeShardPolicy 时间分片**：按日期自动生成表名 Table_yyyyMMdd
- **M4-3 主键/哈希分片**：按主键范围或哈希值路由到不同表
- **M4-4 分片透明路由**：业务层代码不变，自动路由到对应分片
- **M4-5 跨分片查询 (AutoShard)**：跨多个分片的聚合查询

## 7. M6 高级数据中间件

- **M6-1 ETL 数据管道**：抽取、转换、加载全流程管道
- **M6-2 数据同步 (Sync)**：跨库数据同步
- **M6-3 数据备份与恢复**：数据库级别备份恢复
- **M6-4 实体队列 / 异步批写**：实体变更异步队列，批量合并写入
- **M6-5 实体拦截器 (AOP)**：插入/更新/删除前后钩子
- **M6-6 脏数据跟踪**：实体属性变更追踪
- **M6-7 树形实体**：支持树结构实体操作
- **M6-8 数据权限**：行级数据权限过滤
- **M6-9 统计与聚合扩展**：同比环比、聚合查询扩展

## 10. 不做什么

- **AOT/NativeAOT 支持**：XCode 深度依赖反射（实体工厂、元数据缓存、动态属性访问器）和动态代码生成（代码生成器 xcodetool），AOT 裁剪/预编译会严重破坏核心架构。保持运行时动态能力优先，**明确不作为追赶目标**。需要 AOT 的场景推荐 FreeSql / SqlSugar。
- **可视化模型设计器**：经评估不复杂但性价比不高。XML 模型编辑已有 XSD 架构（`Model202604.xsd`）提供智能感知，设计器带来的边际收益有限。**明确不做**。
- **NoSQL 原生查询**：MongoDB 等 NoSQL 数据库仅作为可选 Provider 接入，不提供非关系型原生查询 API
- **GraphQL 支持**：超出数据中间件定位范围
- **微服务/服务网格**：不内置服务发现、负载均衡等，由 NewLife.Stardust 生态提供
